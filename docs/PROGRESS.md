   seconds, combined with the frontend's existing `disabled={submitting}` button
   state. Frontend `uuid.ts` deleted. DTO/tests/docs updated. See PROGRESS.md

   strategy), testing, project structure, and "what I'd improve". Replaced the
   NestJS/Vite boilerplate sub-READMEs with concise project pointers.
   Commit `a9485ba`.

3. **`GET /bookings?eventId=<unknown>` returned a silent empty 200 (§8).**
   Indistinguishable from "event exists but has no bookings".
   → Now verifies the event exists and returns **404** with a clear message,

   `e7c938f`.

4. **`GET /bookings/:id` with a malformed id returned a 500 (§8).** A non-UUID
   reached Postgres and raised an invalid-uuid error → generic 500.
   → Added `ParseUUIDPipe` so a malformed id is rejected with **400**. Commit
   `f55f761`.

5. **Booking form validation was too weak (§11).** It relied on permissive
   native `type=email`/`min` and silently rewrote bad seat values.
   → Added explicit client-side validation (event/name/email/seats) with inline
   field errors that block submission before the request is sent. Commit
   `017f106`.

6. **`requestId` was regenerated on every submit attempt (§11).** A retry after a
   failed submit could create a duplicate booking.
   → Generate the `requestId` once per booking and rotate it only after a
   successful submit; added a `safeUuid()` fallback so booking creation still
   works outside secure contexts. Commit `017f106`.

7. **A newly created booking was invisible on page 2+ or under an excluding
   filter (§10/§11).** The success message referenced a booking the user could
   not see, and its live settle was dropped.
   → On create, reset to page 1, clear filters, and optimistically insert the new
   booking so it is always shown and its status settles live. Commit `0fb5012`.

8. **Scoped polling stopped on any transient error and had no upper bound (§12).**
   A single network blip left the badge stuck on PENDING; a never-settling
   booking would poll forever.
   → Poll through transient errors (not a terminal state), bounded by a
   max-attempt cap. Commit `0fb5012`.

9. **Initial paint flashed the empty state before loading (§10).**
   → Initialise `loading=true` so the first paint shows the loading state.
   Commit `0fb5012`.

10. **Custom `Select` was mouse-only (§13 quality).** Options were `<li onClick>`

    → Added `listbox`/`option` ARIA roles, `aria-expanded`/`aria-invalid`, full
    keyboard operation (Arrow/Enter/Space/Escape), and an inline error state.
    Commit `35b7144`.

11. **Booking reference field named `id` in the API response, ambiguous with
    `requestId`.** The booking's own identifier was already backend-generated
    (via `@PrimaryGeneratedColumn('uuid')` in the entity) and the frontend never
    sent or generated a booking ID, but the response field name `id` did not
    make the distinction obvious.
    → Renamed the response field to `bookingReference` in `BookingResponseDto`
    so the API contract explicitly separates:
      - `bookingReference` (UUID, server-generated, the booking record's PK)
      - `requestId` (client-generated idempotency key, sent in the request body)
    Updated frontend types, components, README, and tests to match. The service
    layer and entity still use `booking.id` internally; only the API surface
    was changed.

## Decisions Log
_(record any judgment calls made on ambiguous spec points)_

- **Decision:** ORM = **TypeORM** (not Prisma).
  **Reasoning:** First-class NestJS integration (`@nestjs/typeorm`), straightforward SQL
  migrations, and fine-grained control over the raw atomic `UPDATE ... WHERE available_seats >= n`
  and `SELECT ... FOR UPDATE` needed for overbooking safety, plus native `CHECK` constraints.

- **Decision:** `requestId` is validated as a **required non-empty string** (max length 200),
  NOT a strict UUID.
  **Reasoning:** The assignment's own sample payload uses
  `"requestId": "7f3c2a10-9b1e-4d5a-8c6f-booking-001"`, whose final segment (`booking-001`) is
  not valid hex, so it is not a real UUID. Enforcing `@IsUUID()` would reject the documented
  sample and fail "it runs by following your README/spec." The spec only calls it a
  "client-generated requestId," so a non-empty string is the correct contract.

- **Decision:** `events.id` = integer identity; `bookings.id` = UUID (server-generated), which
  doubles as the **booking reference** (`bookingReference` in the API response).
  `requestId` is a separate client-generated idempotency key.
  **Reasoning:** The sample payload uses `"eventId": 1` (an integer). A UUID booking id is a
  natural, non-guessable, shareable reference and avoids leaking booking counts. Exposing it
  as `bookingReference` in the JSON response makes the distinction from `requestId` explicit.

- **Decision:** Overbooking prevention = **atomic conditional UPDATE** inside a transaction:
  `UPDATE events SET available_seats = available_seats - :n WHERE id = :id AND available_seats >= :n`,
  backed by a `CHECK (available_seats >= 0)` constraint. `rowCount === 1` ⇒ CONFIRMED;
  `rowCount === 0` ⇒ FAILED (sold out).
  **Reasoning:** The row-level write lock Postgres takes during UPDATE serializes concurrent
  updates to the same event row, and the `WHERE ... >= :n` guard makes negative availability
  impossible without needing an explicit advisory/pessimistic lock. Simpler and provably safe.
  (To be implemented + proven with a failing-first concurrency test in Phase 5.)

- **Decision:** Idempotency = `bookings.request_id` **UNIQUE** constraint + BullMQ `jobId = requestId`.
  **Reasoning:** Duplicate detected at insert time (unique violation ⇒ return existing booking)
  and at enqueue time (same jobId is de-duplicated by BullMQ), so a repeated `requestId` never
  creates a second booking or a second job. (Implemented + proven in Phase 5.)

- **Decision:** Retry/backoff = BullMQ `attempts: 3`, exponential backoff base `1000ms`.
  Transient errors (DB connection blips) throw and are retried; business outcomes (event not
  found, sold out) mark the booking FAILED and return **without** throwing (no retry).
  **Reasoning:** Retrying a business failure is pointless and would waste attempts; only
  infrastructure blips deserve a retry.

- **Decision:** Pagination defaults `page=1`, `limit=20`, `limit` capped at `100`.
  **Reasoning:** Sensible, common defaults; cap prevents unbounded result sets.

- **Decision:** A duplicate `requestId` is handled **idempotently** (HTTP 202 with the
  original booking), not with a 409 Conflict.
  **Reasoning:** `requestId` is an idempotency key; the correct semantic for replaying an
  idempotent request is to return the same result, not an error. The plan lists 409 as
  "if applicable" — with idempotency it is not.

- **Decision:** UUID generation moved fully server-side; requestId is no longer client-generated.
  **Reasoning:** Per project decision, all UUID generation was moved from the frontend
  to the backend for consistency and simplicity. The frontend previously generated
  `requestId` via `safeUuid()` (`crypto.randomUUID()` with a fallback) and sent it with
  every booking request; now the controller generates it via `crypto.randomUUID()`.
  **Tradeoff acknowledged:** This deviates from the assignment's literal wording which
  specifies `requestId` as a "client-generated" idempotency key. The UNIQUE constraint
  on `request_id` can no longer catch client-side duplicates (since every request gets a
  unique UUID). The replacement duplicate-protection mechanism is **request fingerprinting**
  (matching eventId + customerEmail + seats within a 30-second window) **combined with**
  the frontend's existing `disabled={submitting}` button state. This two-layer approach
  is genuinely solid: the button catches the most likely double-click, the fingerprint
  catchs near-simultaneous submits from multiple tabs, and the atomic seat deduction in
  the worker prevents overbooking even if a duplicate somehow slips through. The worst
  case is an extra seat booked — never negative availability or corruption.
